Local HNS Diagnostic Lab

Open a native site, then use ☰ → Settings → View diagnostics, Resolver trace, or TLSA / DANE Inspector to confirm HNS proof, delegated nameserver attempts, DNSSEC, DANE, and DoH fallback behavior.

Verified Direct / DANE Checks

Confirm the actual resolution path
After opening a site, inspect Resolver trace for HNS proof status, resource records, nameserver address source, A/AAAA answers, TLSA records, DANE decision, and whether WebPKI or DoH compatibility was used.

Delegated HNS Nameserver Checks

Check delegated authoritative DNS
For names with NS/DS records, confirm the app found usable GLUE/SYNTH or hydrated nameserver addresses, attempted authoritative UDP/TCP 53, and accepted only DNSSEC-secure answers.

Compatibility / Strict Mode Checks

Identify fallback reasons
If compatibility mode loads a site after direct delegated resolution fails, Resolver trace should show the exact fallback reason and whether the HNS DoH answer carried authenticated data.
Compare strict and compatibility modes
Strict mode should never use third-party HNS DoH. If a site fails there, the error should state whether proof, nameserver transport, DNSSEC, TLSA/DANE, or origin address selection failed.

Make My HNS Domain Work

Common fixes:
  • Direct site: add SYNTH4 or GLUE4 with a reachable address.
  • Delegated site: add NS plus usable GLUE4/GLUE6, then serve authoritative DNS on UDP/TCP 53.
  • Strict DANE: add HNS DS, DNSSEC-signed zone data, and TLSA for _443._tcp.name.
